## 关键发现

### 1. Neo4j Document Intelligence 对 HugeGraph 飞书采集的降维打击

Neo4j 6月1日在Aura平台发布 Document Intelligence 预览版，核心能力：
- **零代码文档→知识图谱**：上传PDF/DOCX，自动生成Schema、抽取实体、构建混合图
- **单Prompt引导抽取**：用户描述关注领域，LLM自动调整抽取策略
- **混合图架构**：词汇层（chunk+embedding）+ 实体层（schema-guided entities）
- **路线图**：将支持API/CLI/MCP端点、规模化到数千文档、可组合架构

**与HugeGraph关联**：我们已在Sprint做的飞书文档采集+图谱构建（54+35 tests）本质上是手工实现同一流程。Neo4j将其产品化为一键操作，HugeGraph需考虑是否在AI场景提供类似的"文档→图谱"一键能力。

**差异化机会**：Neo4j Document Intelligence仅限Aura云平台，HugeGraph可提供**开源、本地部署、可定制**的文档→图谱方案，且天然支持大规模（60亿点边验证）。

### 2. 代码知识图谱方向已进入爆发期

5月GitHub两个代码图谱项目（CodeGraph + Understand-Anything）同时冲上热榜，合计3.5万Star，标志着代码图谱从学术走向工程化。

**关键数据**：CodeGraph在7个真实仓库上实现：成本降低35%、Token减少57%、工具调用减少71%。

**与HugeGraph关联**：我们已有代码图谱PoC（poc/0609-code-graph-dual-agent），使用Tree-sitter AST→函数调用图→Gremlin查询。CodeGraph用SQLite存储， Understand-Anything用JSON文件——两者都不是真正的图数据库。**HugeGraph在代码图谱场景的差异化在于：真正的图存储支持复杂遍历+OLAP分析**。

**建议动作**：升级代码图谱PoC，对比SQLite vs HugeGraph在代码分析查询上的性能差异。

### 3. Neo4j MCP Server 持续迭代，但工具数仍为4个

Neo4j MCP Server 6月10日更新到v0.46，主要修复schema兼容性问题和依赖升级。当前仍只有4个工具（get-schema, read-cypher, write-cypher, 新增工具待确认）。

**与HugeGraph关联**：我们设计的MCP Spec有10 Tools + 3 Resources，包含Gremlin统一入口、内置Text2Gremlin、OLAP Vermeer等Neo4j没有的能力。**MCP能力差距正在拉大，HugeGraph应加快推进MCP Server落地**。

### 4. MemGraphRAG 开源确认三层记忆架构的价值

KDD 2026的MemGraphRAG在6月2日正式开源（GitHub代码），三层架构（Schema+Fact+Passage）与冲突检测机制，与我们6月11日的PoC（poc/0610-magma-four-graph-memory→memgraphrag_three_layer.py 10/10 PASS）高度吻合。

**说明我们的技术判断是正确的**：三层记忆+冲突检测是GraphRAG+Agent Memory的明确趋势。HugeGraph已在Sprint 1-10中覆盖实体消解、图谱质量、知识时效等基础能力。

### 5. ArcadeDB对比报告揭示HugeGraph性能短板

ArcadeDB（Neo4j开源替代）发布详细对比，LDBC基准显示HugeGraph/Vermeer在某些算法上与ArcadeDB有显著差距（WCC慢30x、CDLP慢18.7x），但PageRank和BFS优于Neo4j。

**与HugeGraph关联**：Vermeer OLAP引擎的性能优化是长期任务。60亿点边生产验证是HugeGraph的独有优势，需要在benchmark中体现**大规模场景下的性能表现**，而非仅对比小规模算法执行时间。

## PoC建议

### Go: 代码知识图谱 — SQLite vs HugeGraph 对比 PoC
- **理由**：代码图谱是当前最热方向（GitHub 3.5万Star），CodeGraph用SQLite证明了预索引的价值，但SQLite不是图数据库。HugeGraph可用Gremlin做更复杂的代码分析（多跳调用链、影响半径分析、框架模式检测）
- **优先级**：P0（高关联度 + 可立即落地）
- **实现**：Tree-sitter解析→HugeGraph存储→MCP工具暴露→对比CodeGraph的SQLite方案
- **对标CodeGraph**：同样7个仓库、同样结构类问题、对比查询延迟和准确性

### Watch: 文档→图谱一键管道（对标Neo4j Document Intelligence）
- **理由**：Neo4j已产品化，HugeGraph需要开源替代方案
- **优先级**：P1（需先完成MCP Server基础能力）
- **依赖**：MCP Server + 文档解析器

### Watch: MemGraphRAG生产化
- **理由**：学术验证完毕，需工程化落地
- **优先级**：P2（已有PoC验证，需等待更多生产场景）

### No-go: 超图供应链风险预测（HG-DRA）
- **理由**：超图神经网络偏学术，与HugeGraph当前工程能力差距较大
- **优先级**：P3（仅跟踪）
